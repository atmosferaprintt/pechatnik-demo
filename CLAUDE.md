# CLAUDE.md — CRM «ПЕЧАТНИК»

CRM для типографии/копицентра «ПЕЧАТНИК» (заказчица — Кристи). Владелец проекта — Рабия (типография Atmosfera, автор PrintCRM / atm-crm.ru). Общение — по-русски, весь UI — по-русски. Проект-образец: `~/PrintCRM` (повторяем подход, НЕ повторяем монолит).

**Дизайн — по [ДИЗАЙН.md](ДИЗАЙН.md)** (стиль Crextio: кремово-жёлтый, пилюли, крупные скругления). Рефы — PNG в корне проекта.

## Стек

- React 19 + Vite, чистый JavaScript/JSX, **БЕЗ TypeScript**
- Supabase (PostgreSQL), клиент `@supabase/supabase-js`
- **Без роутера** — навигация вкладками через `useState` + `localStorage` (ключ `tab`)
- **Без UI-библиотек** — inline-стили через style-объекты, эмодзи как визуальные маркеры
- PDF: `jspdf` + `html2canvas`; Excel: `xlsx`
- Деплой: Docker (node build → nginx) на VPS через `scripts/deploy.sh` (rsync)

```bash
npm run dev      # dev-сервер Vite с HMR
npm run build    # прод-сборка → dist/
npm run lint     # ESLint
npm run preview  # локальный просмотр прод-сборки
```

## Архитектура — с первого дня

Главное отличие от PrintCRM: **не растим монолит**. В PrintCRM `App.jsx` дорос до ~27k строк — здесь так нельзя.

- `src/App.jsx` — **только каркас**: авторизация, загрузка данных (`loadAll`), навигация по вкладкам. Новый код в него не добавляем, только 2–3 строки на подключение секции.
- **Каждый раздел — отдельный файл в `src/sections/`.** Один файл = один раздел, PascalCase, default export.
- Supabase-клиент создаётся **один раз** в `App.jsx` и передаётся в секции пропсом. Не дублировать `createClient`.
- `currentUser` и права — тоже пропсами. Никакого своего auth в секциях.
- Стандартные пропсы секции: `supabase`, `currentUser` / `userRole`, `showToast`, `onUpdate` (колбек обновления данных родителя).
- Раздел разросся >1000 строк → превращаем в папку `sections/ИмяРаздела/` с `index.jsx` и подкомпонентами.
- Мобильная версия (если будет) — отдельное дерево `src/mobile/` со своим `useIsMobile()`.

## Правила работы с Рабией

1. **Сначала план — потом код.** Перед любыми изменениями выписать план (что меняем, в каких файлах, риски, открытые вопросы) и ждать явного «ок / утверждаю / делай». Исключение — тривиальная правка по прямой однозначной команде («поменяй текст кнопки на X»).
2. **UI-редизайны:** сначала HTML-макет с 2–4 вариантами (файл `*-variants.html` в корне проекта), Рабия выбирает — потом код.
3. **Деплой только по команде Рабии.** После успешного деплоя — `git commit` (Conventional Commits по-русски: `feat: …`, `fix: …`) + `git push`, в ответе указать хеш коммита. Деплой упал — не коммитить, разбираться.
4. **Задачи «на потом» — в `TODO.md`**, не выполнять сразу. Команды «запланируй задачу на будущее / добавь в todo / отложи» → записать по формату из шапки TODO.md и подтвердить одной строкой. «Что у меня в TODO?» → показать активные задачи. Выполненную задачу удалять из файла в том же коммите, что и реализация.
5. **Не просить подтверждения через модалки/`confirm()` в UI** без необходимости.

## Ловушки Supabase (из опыта PrintCRM)

- `.select()` отдаёт максимум 1000 строк — с первого дня использовать хелпер `loadAllRows(table, columns, pageSize=1000)` с пагинацией через `.range()`.
- **RLS:** решение зафиксировать до первой таблицы (в PrintCRM RLS выключен, права проверяются в коде на клиенте). Пока не решено — спросить Рабию.
- **Миграции — руками через SQL**, файлы миграций вести в репо (`supabase/migrations/` или `db/`), нумеровать по дате.
- Realtime-подписки через `.on('postgres_changes', …)` для живых обновлений.
- Телефоны нормализовать до 10 цифр для сопоставления (как `normalizePhone()` в PrintCRM).

## Соглашения по коду

- camelCase для переменных/функций, PascalCase для компонентов, CONSTANT_CASE для констант
- Тогглы состояния: префикс `show*` (`showAddDeal`), булевы — `is*`
- Модалки — inline условным JSX: `{showModal && (…)}`
- Цветовые карты — константами: `STAGE_COLORS`, `ROLE_COLORS_MAP` и т.п.

## Состав первой версии (утверждено 2026-07-13)

Разделы: **Задачи** (канбан), **Клиенты**, **Финансы**, **Аналитика** (только владелец), **Настройки** (только владелец).

Ключевые решения:
- Этапы канбана: **Новая → В работе → Производство → Готово**
- Приходы фиксируются в Финансах; менеджер связывает приход с задачей (`transactions.task_id`), не наоборот
- 4 сотрудника вносят операции «ввёл и провалился»: не видят итогов, чужих записей и приватных категорий. Полный доступ — только у владельца (Кристи), везде виден автор записи (`created_by`)
- Способ оплаты у дохода: нал / СБП / карта / безнал / перевод; у перевода указывается банк-карта (справочник `banks`, 4–5 карт)
- Категории доходов И расходов владелец добавляет сам в Настройках (`categories.kind`: income / expense_shared / expense_work / expense_personal)
- «Уход» из старого Excel = обычный расход, отдельного типа нет
- **Личные расходы Кристи** (`expense_personal`) — полностью отдельно от бизнеса (решение 2026-07-14): в форме расхода у владельца переключатель «Рабочий / 🔒 Личный», личные не входят в «Расходы за день», общую ленту, расходы и прибыль в Аналитике; показываются своим блоком (жёлтая карточка в Финансах, карточка в Аналитике, блок «ЛИЧНЫЕ» в Excel-экспорте). Наличные личные траты в расчёте кассы учитываются (деньги физически ушли из кассы)
- Сверка дня: расчётный счёт подгружается **через API банка** (`bank_statements`, как в PrintCRM), наличка вводится вручную; строки выписки без привязанной операции = подсветка «пришло, но не записано»
- RLS: включаем минимум на transactions / day_closures / categories (сотрудники не должны достать итоги через прямой API-запрос) — политики отдельной миграцией

Схема БД: [supabase/migrations/001_init.sql](supabase/migrations/001_init.sql).
